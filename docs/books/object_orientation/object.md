プログラミングをより抽象的で現実世界に近いモデリングの下で扱う、という動機がある？
基本的には、オブジェクトは
- 状態と振る舞いを同時に閉じ込める...抽象データ型的な発想
- 各オブジェクトは相手にメッセージを送る...メッセージパッシング的な発想
- 各オブジェクトの一面的な見方のみを用いる...クラスベースとかインターフェースとか

## 具体例

### 関数テーブルの例
パッと出てくる例がわかりにくい...

いろんな種類の物体が複数表れる画像を書く。
球体・多角形・表面がベジェ曲線で書かれている...
みたいなのを考えることになる。
```
struct Sph { ... }; struct Poly { ... }; struct Bj { ... };
fn hit_sph(sph: Sph, ray: Ray) -> color { ... }
fn hit_poly(poly: Poly, ray: Ray) -> color { ... }
fn hit_bj(bj: Bj, ray: Ray) -> color { ... }
```

シーン全体を表すには？
```
fn render(sphs: Vec<Sph>, polys: Vec<Poly>, bjs: Vec<Bj>) -> Image {
  for s in sphs {
    ...
  }

  for p in polys {
    ...
  }
}
```
データが増えれば大体全ての関数を書き直すことになるので、めんどくさい。
直和型があれば、ある程度いい感じ。
```
enum Obj {
  Sph, Poly, Bj,
}

fn hit_obj(obj: Obj, ray: Ray) -> color { ... }

fn render(obj: Obj) -> {
  ...
}
```
いくつか不満がある："可能なオブジェクト"を事前に列挙する必要がある。
というか、あるオブジェクトに対応する `hit` 関数があればいいだけなのに、可能なオブジェクトを全部列挙する？
実行時にどの関数を呼ぶかを保存しておくと、そもそも束ねる必要がなくなる：エラーは実行時に出るようになるかも。
```
struc Sph {
  center: (f32, f32, f32),
  radius: f32,
  call_render: fn (Sph, Ray) -> color,
}

fn render(obj: ?) -> {
  let color = (obj.call_render)(obj);
}
```
すごいことになってるが、意味はわかる。

### 現実世界（？）の例
次々とこんなデータが入ってくる。
```
{
  name: "test",
  comment: "これが通知に出たらバグ",
}
{
  full_name: "中学 太郎",
  age: 25,
}
{
  first_name: "高校",
  family_name: "二郎",
  age: 25,
}
```
名前を扱うとすると、ここには境界が存在して、その前後ではこういうことをする。
- その人（？）の情報を見て名前を作り上げる。
- 作り上げた名前をもとに他の手続きを組む。

この分け方ができるなら、実装は別々に行うことができる。
これはつまり、「_名前が取り出せる_ なら、**それの実体がなんであろうと**、この後の処理は全部行える」"境界"が見えている。
境界のイメージとしては窓口を考えるといい。
成年・未成年だろうと法人だろうと、名前があるならその窓口で正しく処理がされるケース、みたいな。

## イメージ
もうちょっと現実に近いような抽象的な議論。
プログラミングを用いて現実のモデリングとして扱いたい対象すべてを包括するような概念をオブジェクトということにする。
何が必要？
- 各オブジェクトはある瞬間に作られて、何 _時間の単位_ か存在して、消えてゆく。
  - 作られた瞬間がなかったり消えることがなかったりしてもよくて、その場合は無限とかを考えればいいと思う。
- オブジェクトを区別したいときがくると思う。前に扱ったオブジェクトなのかどうか。
- "オブジェクト間"の呼び出しの場合はその具体的な実行内容は向こうに任せる。

```
Object Sph {
  center: (f32, f32, f32),
  radius: f32,
  render: Color -> Sph {
    // 当然、**この**オブジェクトをどう使うかを書く。
  }
}

// どんなデータが来ようと、やること**の記述**は同じ。
fn render(obj) -> Image {
  for i in 0..n {
    let color = obj.render( ray(i) )
  }
}
```

これをもっと過激にして、すべてをオブジェクトにしてしまうのがオブジェクト指向。
例えば、上のプログラムの `fn render()` はどのオブジェクトにも属していないので、
これを"プログラム"というオブジェクトに属していると考える。
```
Object Program {
  render: { ... }
}
```
あと、整数もオブジェクトにする。

## GC について
オブジェクト指向を実現するにあたり、あるオブジェクトがいらなくなったかどうかの判定を自動的に行いたくなる。
これを行うことを garbage collection という。
GC は実装側のものだが、「こう実装されているからこう書くと一瞬止まる」みたいなことが起きるので、
ユーザー側に実装側の話がある程度共有されてもいる。

だいたいオブジェクトは heap に確保されるので、実装側が `free` を使って適切にオブジェクトのライフサイクルを管理する。

### reference とか reallocation について
例えばオブジェクトを勝手に大きくすると、実装側としては heap 上のオブジェクトの位置を変える必要が出てきたりする。そうなると、そのオブジェクトを利用している他のものがアップデートを行う必要が出てきたりする。
それを避ける実装の都合上とかで、直接 reference をとらずに一度何かの store に問い合わせる経由でアクセスをするようにできる。 
ついでに、使われなくなったオブジェクトを勝手に破棄したりできるといいので、
オブジェクト指向言語というと、後ろに何かしらのランタイムみたいなのがあって、オブジェクトの生存を管理しているイメージがある。
