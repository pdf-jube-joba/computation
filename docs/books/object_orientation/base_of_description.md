## 結合と interface の話
あるオブジェクトに無造作に大量のメソッドがあるのではなくて、各目的ごとにメソッド群をまとめることができるといい。
各オブジェクトの持つ様々な側面を抽出することを考える。
例：「動物であるなら○○ができる」と「社会的な人間であるなら××ができる」の両方を、1つのオブジェクトから使い分けるとして、
単に○○・××に対応するメソッドが並列に備わっているとは考えずに、「～であるなら」を一段階挟むことができると、読みやすくなったりする。

1. prototype based / class based
  オブジェクトの作り方として、すでに存在するオブジェクトをもとに作る/型枠みたいなのから作る、みたいな分け方。
  prototype based はさておき、 class based はこれから作るオブジェクトが、まず何の一種なのかを決定してから作ることになる。
  実在するものを分類するのと同じ？
  オブジェクトの様々な側面をとるということで、「○○の一種ということは××の一種でもあって、××と同様に振る舞う」とかもできるとうれしい。
2. type class や trait
  生のオブジェクト（未分類）みたいなものがあって、
  オブジェクトに対して、各側面ごとに「○○として動く場合はこのように動く」というのを宣言する方法に近い？
  （もちろん生のオブジェクト自体が型付けされていることはあるだろうけれど。）

ここで、相手に自分自身の情報を渡すときに、全ての情報を渡さずにその側面としての情報だけわたっていれば大体は十分なはず。
例えば、呼び出しの情報だけ見ていると人間であることが要求されているのに、
その関数の中で実質的に成年であることが要求されるような手続きがあったとしたら、
それは手続きの要件を定義した側がおかしい（どんな側面が必要なのかを外部に十分に伝えることができていない。）
